1. Overview

PM Lens は Claude Code 用のプロジェクト管理 MCP (Model Context Protocol) サーバー。 タスク追跡・進捗可視化・ブロッカー検知・ADR 記録・横断ダッシュボード・workflow 状態マシン・ SQLite メモリ層 (FTS5 全文検索) を一体提供する。

言語Python 3.11+
主要フレームワークFastMCP, Pydantic v2, Click, Jinja2, tomlkit
データ形式YAML (human-readable, git-friendly) + SQLite per-project (.pm/memory.db) + global (~/.pm/memory.db) + outbox (~/.pm/desktop/desktop.db)
配布PyPI pmlens (Full RW) + .mcpb bundle (Lens RO / Desktop Outbox)
対象ホストClaude Code, Claude Desktop, Claude Cowork, Codex CLI
動作モードPM_LENS=0 Full RW / PM_LENS=1 Lens RO / PM_LENS=1 + PM_DESKTOP_WRITE=1 Outbox bridge (Phase 2)
ライセンスMIT

2. Module Architecture

18 個の Python モジュールが server.py (FastMCP インスタンス) を中心に連携する。 データ層 (models / storage / memory / outbox) が下位、機能層 (workflow / hooks / installer / discovery / velocity / dashboard / recall / rules) が中位、orchestration (server / __main__) が上位。 outbox.py は v0.8.0 で追加された Phase 2 Desktop Outbox SQLite store (ADR-019)。

点線 (utils.py) は多数モジュールから利用される依存。discovery.py は深さ 5 cap + 除外 ディレクトリ + ~/.pm/ スキップで bounded walk を保証 (PMSERV-081)。 memory.py は PM_LENS=1 時に ?mode=ro&immutable=1 で SQLite を開き WAL sidecar の意図せぬ作成を防ぐ (PMSERV-080)。

4. Lens Mode (Read-Only) Flow

PM_LENS=1 環境変数が立った PM Lens プロセスは、起動時に RO_ALLOWLIST 外のツールを FastMCP に登録しない。これは Claude Desktop / Cowork のような passive ビューア host で「書き込み事故」を 構造的に防ぐ防衛機構。

構造的強制 vs ポリシー宣言: manifest の server.mcp_config.env.PM_LENS=1 は配布意図のシグナル (host が読まなければ無効) だが、RO_ALLOWLIST による FastMCP 未登録は code レベルの構造的不変条件。 1 箇所のロジック漏れがあっても他方で防御するレイヤード設計 (ADR-015/017/018)。

5. Workflow Engine

5 つの builtin テンプレートでステートマシン化された開発プロセスを提供。各 step が tool_hint/skill_hint/agent_hint でガイダンスを返し、 gate: user_approval step では明示的承認待ち、loop: true step では 品質スコアによるループバックを許容する。

5.1 development (9 steps)

5.2 super-research (6 steps, depth_check ループ付き)

5.3 discovery (調査・ブレスト)

scope → research → analysis → synthesis → proposal の 5 step。完了時に chain_to: development で次の開発 workflow 起動を提案する。

5.4 brainstorming (8 steps, diverge↔evaluate ループ)

3 並列エージェントで発散 → 6 dimension scoring で収束 (Double Diamond)。要件・仕様を KR 化し ADR 起票して chain_to: development。詳細は workflow-guide.html §5.4。

5.5 content-pipeline (4 steps, ADR-024)

.pm の知見 (lesson/insight/ADR/defect) を redact 済みの build-in-public X 下書きへ変換 (extract → draft → redact → review)。PM Lens は X 認証情報を持たず投稿しない — redaction が唯一の安全層。 詳細は workflow-guide.html §5.5。

6. Task & Issue Lifecycle

タスクは todo → in_progress → review → done の主経路、別途 blocked (任意状態から推移可) を持つ。 issue (子 task) を severity=defect で起票すると親 task が自動的に done → review に巻き戻る。

7. Data Layout

YAML (human-readable, git-friendly) + SQLite (高速検索) のハイブリッド。プロジェクトローカル .pm/ と global ~/.pm/ が分離。

Per-project (<repo>/.pm/)

.pm/ ├── project.yaml # メタデータ + phases + pm_schema ├── tasks.yaml # 全タスク (parent_id で issue 親子) ├── decisions.yaml # ADR-NNN ├── workflows.yaml # WF-NNN 状態マシン ├── knowledge.yaml # KR-NNN (research/spec/etc) ├── memory.db # SQLite + FTS5 (per-project) ├── daily/ │ └── YYYY-MM-DD.yaml # pm_log エントリ └── registry.yaml # 内部メタ (project 自身の参照)

Global (~/.pm/)

~/.pm/ ├── registry.yaml # 登録 project 一覧 (path + name) ├── memory.db # cross-project SQLite (sync 先) └── <cache files> # dashboard cache 等 # discover_projects は ~/.pm/ を skip # (depth-5 cap + excluded dirs と二重防御)

Lens モード時の RO 担保

レイヤ挙動 (PM_LENS=1)
YAML filesread-only open (load_* のみ呼ばれ save_* は到達不能 — RO_ALLOWLIST 外)
per-project SQLite?mode=ro&immutable=1 で接続 — WAL -shm sidecar 不生成 (PMSERV-080)
global SQLiteMemoryStore(readonly=True) で global_db_path=None 強制 — sync 経路無効化
memory.db 不在:memory: SQLite に fallback (search は空、エラーなし)

8. Distribution Channels

同一コードベースが 3 つの配布モードを持つ。PyPI は Full RW、.mcpb は構造的 RO (Lens)、 .mcpb + PM_DESKTOP_WRITE=1 は Lens に Outbox 書込みだけを足した bridge モード。

同一バイナリ?はい — 配布物は 1 種類、env で role gating←
バージョン同期pyproject.toml が source of truth、manifest.json は build_mcpb.py validator で lockstep (CI 強制)
共有 surface の安全性: 両者は ~/.pm/ と project 内 .pm/memory.db を共有しうるが、 Lens 側は memory.py:98-117 の ?mode=ro&immutable=1 で RO 接続するため、 Claude Code 側が WAL を書いていても -shm sidecar の意図せぬ生成は発生しない (PMSERV-080 / FINDING-J)。

installer.py の PM_LENS / PM_DESKTOP_WRITE 認識

install_claude_code() は --env PM_LENS=1 (+ PM_DESKTOP_WRITE=1) を claude mcp add に注入、install_codex() は TOML inline_table env = { PM_LENS = "1" } を section に書き込む。 InstallResult.lens_mode / InstallResult.desktop_write で caller が role を判別可能。 Phase 2 (PMSERV-100) で .mcpb manifest が PM_DESKTOP_WRITE を auto-propagate するため、 Desktop ユーザーは bundle drag & drop だけで outbox bridge が成立する。

9. Desktop Outbox (Phase 2 — ADR-019)

Claude Desktop (Lens) は本来 RO だが、PM_DESKTOP_WRITE=1 時に限り ~/.pm/desktop/desktop.db という専用 SQLite へ append-only で書き込める。 main の per-project .pm/memory.db は依然不可触 (Lens 不変条件保持)。 書込みは「outbox」として一旦溜まり、Claude Code 側で pm_outbox_pending → pm_outbox_merge / pm_outbox_reject でレビューして promote する。
